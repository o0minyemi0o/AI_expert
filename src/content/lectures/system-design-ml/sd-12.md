# 모의 설계 면접

## 왜 모의 설계 면접이 중요한가

ML 시스템 설계 면접은 코딩 인터뷰와 달리 **명확한 정답이 없습니다**. 동일한 문제에 대해 완전히 다른 설계가 모두 "좋은 답"이 될 수 있습니다. coi-12에서 코딩 인터뷰의 모의 면접을 다뤘다면, 여기서는 시스템 설계 면접에 특화된 실전 전략을 다룹니다.

> **핵심 직관**: 시스템 설계 면접에서 떨어지는 가장 흔한 이유는 "기술적으로 부족해서"가 아니라 "시간을 잘못 배분해서"입니다. 45분 안에 전체를 커버하는 페이싱이 핵심입니다.

## 1. 시간 배분 전략

```
ML 시스템 설계 면접 시간 배분 (45분):

  [0-5분] 문제 정의 & 요구사항 (sd-01)
  ├─ 비즈니스 목표 확인
  ├─ 규모, 지연시간, 데이터 가용성 질문
  └─ 범위 좁히기 ("추천 시스템 중 후보 생성에 집중하겠습니다")

  [5-10분] 메트릭 정의
  ├─ 오프라인: AUC, NDCG, RMSE 등
  └─ 온라인: CTR, 체류 시간, 매출 등

  [10-25분] 시스템 설계 (핵심!)
  ├─ 전체 아키텍처 (하이레벨 다이어그램)
  ├─ 데이터 파이프라인 (sd-09)
  ├─ 피처 엔지니어링
  ├─ 모델 선택과 학습
  └─ 서빙 아키텍처 (sd-10)

  [25-35분] 딥다이브
  ├─ 면접관이 관심 있는 영역에 깊이 들어가기
  ├─ 트레이드오프 논의
  └─ 엣지 케이스 처리

  [35-45분] 개선과 확장
  ├─ 모니터링, A/B 테스트
  ├─ 스케일링 전략
  └─ 향후 개선 방향
```

## 2. 트레이드오프 논의 프레임워크

면접에서 가장 높은 평가를 받는 것은 **트레이드오프를 구조화하여 논의하는 능력**입니다.

```
트레이드오프 논의 템플릿:

  "이 문제에는 두 가지 접근이 가능합니다.

   접근 A: [구체적 설명]
   - 장점: [1], [2]
   - 단점: [1], [2]

   접근 B: [구체적 설명]
   - 장점: [1], [2]
   - 단점: [1], [2]

   현재 요구사항(규모/지연시간/데이터)을 고려하면
   접근 A가 더 적합합니다. 이유는 [구체적 근거]입니다.
   다만, 향후 [조건 변화] 시 접근 B로 전환을 고려해야 합니다."
```

| 트레이드오프 축 | 옵션 A | 옵션 B | 판단 기준 |
|-------------|--------|--------|----------|
| 모델 복잡도 | 단순 (LR, 트리) | 복잡 (DNN) | 데이터량, 지연시간 |
| 서빙 방식 | 배치 | 실시간 | 신선도 요구사항 |
| 피처 저장 | 사전 계산 | 실시간 계산 | 비용, 신선도 |
| 학습 방식 | 전체 재학습 | 증분 학습 | 데이터 변화 속도 |
| 모델 수 | 단일 글로벌 | 사용자별 | 데이터량, 개인화 |

> **핵심 직관**: "A가 좋습니다"보다 "이 상황에서는 A가 적합하지만, 규모가 10배 커지면 B를 고려해야 합니다"가 훨씬 높은 평가를 받습니다. sd-01에서 다룬 프레임워크를 여기서 실전에 적용합니다.

## 3. 실전 시뮬레이션: 콘텐츠 모더레이션 시스템

**문제**: "대규모 소셜 미디어 플랫폼의 콘텐츠 모더레이션 시스템을 설계하세요."

```
[0-5분] 요구사항 명확화:

  "몇 가지 질문을 드리겠습니다.
   - 어떤 종류의 콘텐츠를 다루나요? (텍스트, 이미지, 비디오)
   - DAU와 일일 게시물 수는?
   - 실시간 차단이 필요한가요, 사후 검토도 괜찮나요?
   - 오탐(정상을 삭제)과 미탐(유해를 놓침) 중 어느 것이 더 심각한가요?"

  → 가정: 텍스트+이미지, DAU 5억, 일일 10억 게시물
          실시간 필터링 필요, 미탐이 더 심각


[5-10분] 메트릭:

  오프라인: Precision, Recall (카테고리별)
  온라인: 유해 콘텐츠 노출률, 오탐으로 인한 항의율
  가드레일: 처리 지연시간 < 500ms


[10-25분] 시스템 설계:

  [게시물 업로드]
       │
       ▼
  [1차 필터: 규칙 + 해시 매칭]
  ├─ 블랙리스트 단어/해시 → 즉시 차단
  └─ 통과 → ML 모델
       │
       ▼
  [2차 필터: ML 분류]
  ├─ 텍스트 분류기 (혐오, 스팸, 폭력)
  ├─ 이미지 분류기 (NSFW, 폭력, 저작권)
  └─ 종합 점수 계산
       │
       ├─ 고위험 (>0.9) → 자동 차단
       ├─ 중위험 (0.5-0.9) → 사람 검토 큐
       └─ 저위험 (<0.5) → 게시 허용
       │
       ▼
  [사람 검토 + 피드백 루프]
  └─ 검토 결과 → 모델 재학습


[25-35분] 딥다이브 (면접관 질문 대응):

  Q: "새로운 유형의 유해 콘텐츠가 등장하면?"
  A: "규칙 엔진으로 즉시 대응(1시간 내 배포) +
      데이터 수집 후 모델 재학습(1-2주)"

  Q: "다국어 환경에서는?"
  A: "sd-07에서 다룬 다국어 NLP 전략 적용.
      주요 언어는 언어별 모델, 나머지는 다국어 모델"

  Q: "오탐을 줄이려면?"
  A: "사기 탐지(sd-06)와 유사하게 비용 민감 임계값 조정.
      카테고리별 다른 임계값 적용"
```

## 4. 흔한 실수와 대응

```
실수 유형                         대응 전략
──────────────────────────────────────────────────────
모델부터 이야기한다              → 요구사항 먼저, 모델은 나중에
"GPT-4를 쓰면 됩니다"           → 비용, 지연시간, 제어 가능성 논의
데이터 파이프라인을 빠뜨린다      → sd-09 프레임워크 적용
모니터링을 언급하지 않는다        → sd-10의 3계층 모니터링
스케일 고려하지 않음             → "DAU 10배 증가 시" 시나리오 논의
단일 접근만 제시                → 반드시 대안과 트레이드오프 제시
```

## 5. 자가 평가 루브릭

| 영역 | Strong Hire | Hire | No Hire |
|------|------------|------|---------|
| 요구사항 | 능동적 질문, 범위 설정 | 기본 질문 | 질문 없이 시작 |
| 메트릭 | 오프라인+온라인+가드레일 | 오프라인만 | 메트릭 미정의 |
| 아키텍처 | 전체 파이프라인 + 깊이 | 주요 컴포넌트 | 모델만 언급 |
| 트레이드오프 | 자발적 논의, 대안 제시 | 질문 시 설명 | 논의 없음 |
| 실무 경험 | 구체적 사례 언급 | 일반적 답변 | 교과서적 답변 |
| 깊이 | 핵심 영역 딥다이브 | 전반적 커버 | 표면적 |

## 6. 복기와 개선 전략

```
설계 면접 복기 템플릿:

  1. 시간 배분
     - 요구사항: __분 (목표: 5분)
     - 설계: __분 (목표: 15분)
     - 딥다이브: __분 (목표: 10분)
     → 어디서 시간을 초과/절약했는가?

  2. 트레이드오프
     - 자발적으로 논의한 횟수: __회
     - 면접관 질문 후 논의한 횟수: __회
     → 트레이드오프를 더 능동적으로 제시할 수 있었는가?

  3. 누락된 컴포넌트
     - 데이터 파이프라인: □ 포함 / □ 누락
     - 모니터링: □ 포함 / □ 누락
     - 스케일링: □ 포함 / □ 누락
     → 체크리스트를 통해 누락 방지

  4. 액션 아이템
     - (예: 추천 시스템 설계 연습 → sd-02 복습)
     - (예: 트레이드오프 논의 연습 → 매일 1개 시스템 분석)
```

> **핵심 직관**: 설계 면접은 "완벽한 설계"가 아니라 "설계 과정에서 보여주는 사고력"을 평가합니다. sd-01에서 배운 프레임워크를 sd-02~sd-11의 구체적 시스템에 반복 적용하면, 어떤 문제가 나와도 체계적으로 접근할 수 있습니다.

이 강의로 system-design-ml 과정을 마무리합니다. sd-01의 프레임워크부터 sd-11의 LLM 시스템까지, 각 도메인별 설계를 학습한 후 이 모의 면접으로 실전 적용 능력을 검증하고 개선하는 것이 최적의 학습 경로입니다.

## 핵심 정리

- 45분 설계 면접에서 **시간 배분**(요구사항 5분, 설계 15분, 딥다이브 10분, 확장 10분)이 합격을 결정합니다
- 트레이드오프 논의는 **"상황에 따라 A가 적합하지만, 조건 변화 시 B를 고려"** 형식으로 구조화합니다
- 흔한 실수는 **모델부터 시작하기, 데이터 파이프라인 누락, 단일 접근만 제시**이며 체크리스트로 방지합니다
- 자가 평가 루브릭으로 **요구사항, 메트릭, 아키텍처, 트레이드오프, 깊이** 5개 영역을 점검합니다
- 복기에서는 **시간 초과 구간, 누락 컴포넌트, 트레이드오프 빈도**를 분석하여 다음 연습에 반영합니다
